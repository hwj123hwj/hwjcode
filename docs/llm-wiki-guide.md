# LLM Wiki 用户指南

> 基于 Karpathy 的 LLM Wiki 模式，让 AI 为你构建和维护项目知识库。

## 快速开始

两步即可：

```
/wiki init
/wiki ingest
```

第一步创建目录结构，第二步把 `raw/` 下所有源文件一次性整理进知识库。

## 核心命令

| 命令 | 作用 |
|------|------|
| `/wiki init` | 初始化知识库 |
| `/wiki ingest` | 将 `raw/` 下所有未整理的源文件批量摄入 |
| `/wiki ingest <文件>` | 摄入指定文件（可以是项目中任意文件，不限于 raw/） |
| `/wiki query <问题>` | 向知识库提问（别名：`/wiki q`、`/wiki ask`） |
| `/wiki lint` | 健康检查（找孤页、矛盾、缺口） |
| `/wiki status` | 查看知识库状态和近期活动 |
| `/wiki log` | 查看完整操作日志 |

所有命令的提示词对用户不可见，你只会看到简短的状态提示，然后 AI 直接开始工作。

## 日常使用场景

### 场景 1：初始化后一键整理

把想学的文件丢进 `.llm-wiki/raw/`，然后：

```
/wiki ingest
```

dvcode 会自动扫描 `raw/` 目录，跳过已经整理过的，把所有新文件逐个摄入。每个文件摄入时会：
- 读完全文，提取关键实体、概念、事实
- 创建一篇源文件摘要页
- 自动更新/创建相关的实体页、概念页
- 更新索引和日志
- 一个文件可能涉及 5~15 个 wiki 页面的创建或更新

### 场景 2：摄入项目中的某个文件

不需要先放进 `raw/`，直接指定路径：

```
/wiki ingest docs/api-design.md
/wiki ingest src/core/event-bus.ts
```

### 场景 3：学一个网页

直接在对话中说：

```
帮我把这篇文章学到wiki里 https://example.com/some-article
```

dvcode 会抓取网页内容，然后按 wiki 流程整理进知识库。

### 场景 4：把对话中的收获存入 wiki

你跟 dvcode 聊了半天某个技术方案，聊出了好东西，直接说：

```
把我们刚才关于 WebSocket 连接池方案的讨论整理到wiki里
```

dvcode 会把对话中的有价值内容提炼成 wiki 页面。

### 场景 5：提问并回写

```
/wiki query 我们项目中所有涉及认证的模块有哪些？它们之间的关系是什么？
```

dvcode 会**只从 wiki 知识页面中查找**答案（不会直接读原始文件）。如果知识库里缺少相关内容，它会告诉你该先 ingest 哪些文件。

回答后它会问你要不要把这个分析也存入 wiki。选是，知识就沉淀下来了。

### 场景 6：定期维护

每隔一段时间跑一次：

```
/wiki lint
```

dvcode 会检查知识库的健康状况——哪些页面是孤岛、哪些链接失效、哪些概念频繁出现但还没有独立页面、哪些内容之间有矛盾。

## 让 dvcode 自动学习的技巧

不需要每次都用 `/wiki` 命令。wiki 初始化后，dvcode 在日常对话中就已经感知到它的存在。用自然语言提到"wiki"就够了：

**直接指令式：**
- "把这个整理到 wiki"
- "学到 wiki 里"
- "更新 wiki 中关于 XX 的内容"

**摄入外部内容：**
- "读一下 `xxx.md`，重点内容整理到 wiki"
- "帮我把这个 URL 的内容学到 wiki 里"
- "把这份 API 文档摄入 wiki"

**对话沉淀：**
- "我们刚才的讨论有用，存到 wiki"
- "把这段分析加到 wiki 对应的页面上"

**批量学习：**
- "把 `docs/` 下所有 `.md` 文件都摄入 wiki"
- "把这5篇论文的核心观点整理到 wiki，注意标注来源和矛盾点"

dvcode 会自动按知识库维护的流程操作：读源文件 → 提取知识 → 更新/创建页面 → 维护索引和日志。

## 目录结构

```
.llm-wiki/
├── raw/              ← 原始素材（不可变，dvcode 只读不改）
├── wiki/             ← 知识页面（全部由 dvcode 维护）
│   ├── overview.md          项目概览
│   ├── source-xxx.md        源文件摘要
│   ├── entity-xxx.md        实体页面（人、项目、组件等）
│   └── concept-xxx.md       概念页面（模式、原理、技术等）
├── index.md          ← 内容索引（导航用）
└── log.md            ← 操作日志（时间线）
```

**你不需要手动编辑 `wiki/`、`index.md`、`log.md`。** 这些全由 dvcode 维护。

你要做的：
- 往 `raw/` 里放原始文件（文章、论文、笔记），然后 `/wiki ingest`
- 或者直接 `/wiki ingest <任意文件路径>` 摄入项目中的文件
- 用 Obsidian 或任何 Markdown 编辑器浏览 `wiki/` 下的页面
- 用 `index.md` 作为知识库的目录导航

## 查询规则

`/wiki query` 有一条重要规则：**只从 `wiki/` 下的知识页面查找答案，不会直接读 `raw/` 里的原始文件。**

如果知识库里没有你问的内容，dvcode 会明确告诉你缺什么，并建议你运行 `/wiki ingest <文件>` 先把相关源文件整理进来。这确保了回答始终基于已整理的结构化知识，而不是临时拼凑。

## 与 Obsidian 配合

wiki 中使用 `[[wikilinks]]` 格式做交叉引用，天然兼容 Obsidian。

推荐工作流：
1. dvcode 在终端里运行，负责写和维护
2. Obsidian 打开 `.llm-wiki/` 目录，负责浏览和可视化
3. 用 Obsidian 的 Graph View 看知识网络的全貌

## 适用场景

- **项目文档沉淀**：把散落在代码注释、README、PR 中的知识整合
- **技术调研**：读论文、读文章，增量构建对某个技术领域的理解
- **团队知识库**：会议纪要、设计决策、踩坑记录，AI 做整理
- **学习笔记**：读书、看课程，逐章摄入，自动构建知识图谱
- **竞品分析**：多个竞品的资料摄入后，自动生成对比和综合分析

## 核心理念

传统做法（包括 RAG）是每次提问都从零检索、重新拼凑答案。LLM Wiki 不同：

- 知识被**预先编译**成结构化页面，不是每次重新推导
- 每摄入一个新源，所有相关页面都会被更新——知识在**复利增长**
- 交叉引用、矛盾标注、综合分析**已经做好了**，不用每次重来
- 你负责找源、提问、思考；AI 负责所有的苦力活
